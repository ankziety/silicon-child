# ADR-0000: Two-Layer Architecture & Schema Rules

## Status
Accepted

## Date
2024-01-01

## Context
The AI-Infant project requires a stable foundation for long-term research data ingestion and processing. We need to establish immovable contracts (schemas) that ensure data integrity while allowing the system to evolve incrementally.

## Decision
We will implement a two-layer architecture with strict schema versioning rules:

### Layer 1: Immutable Contracts (Schemas)
- **DocV1**: Document ingestion contract
- **TraceV1**: Execution trace contract  
- **JobV1**: Job ledger contract

### Layer 2: Processing Pipeline
- Components that operate on validated data
- Can evolve independently of schemas
- Must maintain backward compatibility with all schema versions

## Schema Rules

### 1. Versioning Strategy
- **Never modify existing schema versions**
- Always add `vN+1` for breaking changes
- Maintain backward compatibility for all active versions
- Deprecation period: 6 months minimum

### 2. Schema Stability
- Schemas are immutable once published
- Fields cannot be removed or renamed
- New fields must be optional with sensible defaults
- Breaking changes require new schema version

### 3. Validation Requirements
- All ingested data must validate against published schemas
- Validation occurs at ingestion boundaries
- Invalid data is rejected with clear error messages
- Schema validation failures are logged as JobV1 entries

### 4. Contract Evolution
- New schema versions must be additive
- Old schema versions remain supported indefinitely
- Migration paths must be provided for major version changes
- Schema documentation must include migration guides

## Architecture Details

### Data Flow
```
Raw Data → Schema Validation → Processing Pipeline → Storage
```

### Schema Validation Points
1. **Ingestion**: Validate against DocV1
2. **Processing**: Validate against TraceV1 for each operation
3. **Storage**: Validate against JobV1 for job ledger entries

### Error Handling
- Validation failures are logged as failed JobV1 entries
- TraceV1 entries capture validation errors with full context
- Failed data is quarantined for manual review

## Consequences

### Positive
- **Data Integrity**: Guaranteed validation at all boundaries
- **Backward Compatibility**: Old data remains accessible
- **Evolution Safety**: System can evolve without breaking existing data
- **Audit Trail**: Complete traceability through JobV1 ledger

### Negative
- **Storage Overhead**: Multiple schema versions require more storage
- **Complexity**: Additional validation layer adds complexity
- **Migration Cost**: Major schema changes require careful migration

## Implementation

### Schema Files
- `schemas/doc.v1.json`: Document ingestion contract
- `schemas/trace.v1.json`: Execution trace contract
- `schemas/job.v1.json`: Job ledger contract

### Validation Framework
- JSON Schema validation using `jsonschema` library
- Pydantic models for runtime validation
- Comprehensive test suite for schema compliance

### CI Integration
- Schema validation in CI pipeline
- Automated testing against all schema versions
- Fixture validation tests

## Future Considerations
- Schema registry for version management
- Automated migration tools for major version changes
- Performance monitoring for validation overhead
- Schema evolution tracking and analytics
